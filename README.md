# BRIGHT Clinical Features DataBase

BRIGHT Clinical Features DataBase is a Python code base for building a structured clinical database from unstructured French neuro-oncology reports. Developed for MVA Master's Hospital Immersion course in collaboration with the BRIGHT team @ Institut de Neurologie, Hôpital Pitié-Salpêtrière. It automates two tasks: **pseudonymization** of patient-identifiable information, and **extraction of 110** standardized clinical and biological features from free-text medical reports.

---

> **⚠️ Current Performance Limitations**
>
> The ML models used in the feature extraction pipeline were trained exclusively on
> **synthetic clinical documents generated by large language models (LLMs)**. These
> documents were produced using few-shot prompting with clinical templates and
> filtered through a 6-level quality cascade, but they are **not real clinical records**
> and **have not been validated against annotated real-world data**.
>
> As a result:
> - Extraction accuracy on real French neuro-oncology reports is **substantially lower**
>   than benchmark numbers suggest.
> - The system should be treated as a **research prototype**, not a clinical-grade tool.
> - Before relying on extracted features for any clinical or scientific analysis,
>   **manual validation on a representative sample is mandatory**.
>
> To improve performance, the models could be retrained on real annotated data.
> See `docs/feature_extraction.md` for how the training data was created and
> `docs/evaluation.md` for how to benchmark extraction quality.

## Installation

### Prerequisites
- conda and pip
- Git
- Hugging Face account (token required for model downloads)

### Automated Setup

**Linux / macOS:**
```bash
conda create -n bright_db python=3.12 && conda activate bright_db
conda config --add channels conda-forge
git clone https://github.com/raphaelrubrice/BRIGHT_ClinicalDataBase.git
cd BRIGHT_ClinicalDataBase
bash scripts/setup.sh
pip install -r requirements.txt
```

**Windows (PowerShell):**
```powershell
conda create -n bright_db python=3.12; conda activate bright_db
conda config --add channels conda-forge
git clone https://github.com/raphaelrubrice/BRIGHT_ClinicalDataBase.git
cd BRIGHT_ClinicalDataBase
Set-ExecutionPolicy -ExecutionPolicy Bypass -Scope Process
.\scripts\setup.ps1
pip install -r requirements.txt
```

### What the Setup Scripts Do
- Install PyTorch, PyMuPDF, EDS-NLP, and other dependencies
- Prompt for your Hugging Face token
- Clone and install `eds-pseudo` (the pseudonymization model)

## Quick Start

### Pseudonymize clinical PDFs
```bash
python main.py pseudo --db path/to/clinical_db.csv --pdfs path/to/pdf_folder/
```
Outputs: `clinical_db_pseudo_only.csv` (text fields replaced with pseudonyms)

To launch the desktop GUI instead:
```bash
python main.py pseudo --gui
```

### Extract clinical features
```bash
python main.py extract --db path/to/clinical_db_pseudo_only.csv --output path/to/output_dir/
```
Outputs: `bio.csv` (biological features) and `clinique.csv` (clinical features)

### Run tests
```bash
pytest src/tests/
```

## Pseudonymization Pipeline

Text is extracted from PDFs via EDS-PDF → PyMuPDF → pypdf, with EasyOCR/Surya-OCR for scanned pages. Each entity flagged for replacement is assigned a deterministic pseudonym: `SHA256(IPP + label + entity_text + salt)[:10]`, stored in a local sidecar file (`<db_path>.pseudonym_salt`). The same entity always maps to the same token within a patient's records.

Not all detected entities are pseudonymized, some are intentionally kept as-is:

- **Pseudonymized:** `NOM`, `PRENOM`, `ADRESSE`, `ZIP`, `VILLE`, `TEL`, `MAIL`, `SECU`. Birth dates (`DATE_NAISSANCE`) are partially masked: day and month are removed, year is kept.
- **Kept as-is:** `IPP` (needed for patient linking), `HOPITAL` (retained for tumour localisation), `NDA`, and clinical dates (`DATE`).
- **Practitioner names** (preceded by Dr, Pr, Professeur, Docteur, or Interne) are kept if the name appears in the BRIGHT team whitelist in `src/database/pseudonymizer.py`; non-whitelisted practitioners are pseudonymized.

Keep the salt file secret and backed up separately, losing it breaks pseudonym consistency across runs. Do not commit it to git. This system has not been reviewed for GDPR compliance; consult a data protection officer before use in a regulated context.

See `docs/pseudonymization.md` for full architecture details.

## Feature Extraction Pipeline

The extraction pipeline reads the pseudonymized CSV and extracts 110 clinical and biological fields per document (55 bio + 55 clinical). For a full list, see the Field Reference table below.

A document classifier routes each document to the appropriate field set (anatomopathology, consultation, RCP, or radiology). Extraction runs in four tiers: (1) regex dates, (2) controlled-vocabulary fuzzy matching, (3) rule-based EDS-NLP patterns, and (4) 10 fine-tuned EDS-NLP CRF models. Rules and ML results are merged: when both agree, confidence is boosted; when they conflict, rules win except for the ~150 fields where the models cross an F1 threshold on the synthetic benchmark.

The timeline builder aggregates values per patient across all documents into two output CSVs: `bio.csv` (one row per surgical event) and `clinique.csv` (one row per consultation).

See `docs/feature_extraction.md` for full pipeline architecture.

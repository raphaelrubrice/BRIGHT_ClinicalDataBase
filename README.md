# BRIGHT Clinical Features DataBase

BRIGHT Clinical Features DataBase is a Python toolkit for building a structured clinical database from unstructured French neuro-oncology reports. It was developed at Institut de Neurologie, Hôpital Pitié-Salpêtrière for the BRIGHT team. The system automates two tasks: pseudonymization of patient-identifiable information, and extraction of 111 standardized clinical and biological features from free-text medical reports.

The project targets clinicians, data scientists, and engineers who work with the BRIGHT neuro-oncology cohort. It handles the full pipeline from raw clinical PDFs to structured CSV output, making it possible to build a longitudinal patient database without manual data entry.

---

> **⚠️ Current Performance Limitations**
>
> The ML models used in the feature extraction pipeline were trained exclusively on
> **synthetic clinical documents generated by large language models (LLMs)**. These
> documents were produced using few-shot prompting with clinical templates and
> filtered through a 6-level quality cascade — but they are **not real clinical records**
> and **have not been validated against annotated real-world data**.
>
> As a result:
> - Extraction accuracy on real French neuro-oncology reports is **substantially lower**
>   than benchmark numbers suggest.
> - The system should be treated as a **research prototype**, not a clinical-grade tool.
> - Before relying on extracted features for any clinical or scientific analysis,
>   **manual validation on a representative sample is mandatory**.
>
> To improve performance, the models must be retrained on real annotated data.
> See `docs/feature_extraction.md` for how the training data was created and
> `docs/evaluation.md` for how to benchmark extraction quality.

## Installation

### Prerequisites
- conda and pip
- Git
- Hugging Face account (token required for model downloads)

### Automated Setup

**Linux / macOS:**
```bash
conda create -n bright_db python=3.12 && conda activate bright_db
conda config --add channels conda-forge
git clone https://github.com/raphaelrubrice/BRIGHT_ClinicalDataBase.git
cd BRIGHT_ClinicalDataBase
bash scripts/setup.sh
pip install -r requirements.txt
```

**Windows (PowerShell):**
```powershell
conda create -n bright_db python=3.12; conda activate bright_db
conda config --add channels conda-forge
git clone https://github.com/raphaelrubrice/BRIGHT_ClinicalDataBase.git
cd BRIGHT_ClinicalDataBase
Set-ExecutionPolicy -ExecutionPolicy Bypass -Scope Process
.\scripts\setup.ps1
pip install -r requirements.txt
```

### What the Setup Scripts Do
- Install PyTorch, PyMuPDF, EDS-NLP, and other dependencies
- Prompt for your Hugging Face token
- Clone and install `eds-pseudo` (the pseudonymization model)

## Quick Start

### Pseudonymize clinical PDFs
```bash
python main.py pseudo --db path/to/clinical_db.csv --pdfs path/to/pdf_folder/
```
Outputs: `clinical_db_pseudo_only.csv` (text fields replaced with pseudonyms)

To launch the desktop GUI instead:
```bash
python main.py pseudo --gui
```

### Extract clinical features
```bash
python main.py extract --db path/to/clinical_db_pseudo_only.csv --output path/to/output_dir/
```
Outputs: `bio.csv` (biological features) and `clinique.csv` (clinical features)

### Run tests
```bash
pytest src/tests/
```

## Pseudonymization Pipeline

The pseudonymization pipeline detects and replaces protected health information (PHI) in the free-text columns of the clinical database CSV. Text is first extracted from source PDFs using a three-level fallback: EDS-PDF (primary) → PyMuPDF → pypdf. For scanned documents (detected by low character yield or high image ratio), an OCR stage is applied: EasyOCR as primary, Surya-OCR as fallback. The resulting raw text is then passed to the EDS-NLP pseudonymization pipeline.

Each detected PHI entity is replaced by a deterministic pseudonym. The pseudonym is generated by hashing the triple `(IPP, entity_label, entity_text)` using SHA256 with a secret salt stored in a local sidecar file (`<db_path>.pseudonym_salt`). This means the same entity always produces the same pseudonym within a patient's records, enabling cross-document entity tracking without exposing real identifiers. The detected entity types are: `NOM`, `PRENOM`, `DATE`, `ADRESSE`, `ZIP`, `VILLE`, `TEL`, `MAIL`, `HOPITAL`, `IPP`, `NDA`, `SECU`.

Healthcare practitioner names (detected via title-prefixed regex: Dr, Pr, Professeur, Docteur, Interne) are handled separately. Members of the BRIGHT team at Pitié-Salpêtrière are listed in a hardcoded whitelist in `src/database/pseudonymizer.py` and are preserved by default in the output — their names are not pseudonymized. Non-whitelisted practitioners are pseudonymized like any other named entity.

The salt file must be kept secret and backed up separately from the clinical data. Without it, pseudonyms cannot be regenerated consistently across pipeline runs. The system is designed for internal research use, not for public data release — it should be reviewed by a data protection officer before use in any regulated context. Do not commit the salt file to version control.

See `docs/pseudonymization.md` for full architecture details.

## Feature Extraction Pipeline

The extraction pipeline reads the pseudonymized CSV and extracts 111 clinical and biological fields per document. Fields span IHC markers, molecular mutations, chromosomal alterations, demographics, treatment dates, surgical details, and clinical evolution. For a full list, see the Field Reference table below.

Extraction uses a hybrid architecture combining rules and ML. First, a document classifier routes each document to the appropriate field set (anatomopathology report, consultation, RCP, or radiology report). A section detector then identifies relevant text regions. Extraction runs in four tiers applied per document: (1) regex date rules covering all date fields, (2) a controlled-vocabulary fuzzy matcher (Find & Check) for categorical fields with a fixed value set, (3) rule-based EDS-NLP patterns for specialized fields, and (4) 10 fine-tuned EDS-NLP CRF models covering all 111 fields across 10 semantic groups. Results from rules and ML are merged: when both agree the confidence is boosted; when they conflict, rules take precedence for fields where rule precision is higher, and the ML result is used for the ~150 fields where the models pass an F1 threshold on the synthetic benchmark.

The timeline builder (`src/aggregation/`) aggregates extracted values across all documents for each patient, producing two flat CSV files: `bio.csv` (biological features, one row per surgical event) and `clinique.csv` (clinical features, one row per consultation event). Multi-event documents are handled by `row_duplicator.py`, which detects and splits rows when a single document references multiple treatment episodes.

See `docs/feature_extraction.md` for full pipeline architecture.
